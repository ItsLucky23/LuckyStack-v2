## Confirmed New Issues

Only **one** new High/Critical finding survived adversarial verification as a genuine, actionable defect.

### 1. Calendar grid days not keyboard navigable — **HIGH** (confirmed)

- **File:** `src/_components/inputs/DatePicker.tsx` (day buttons ~lines 154–165)
- **Detail:** Calendar day cells are native `<button>` elements with only `onClick` / `onMouseEnter` handlers — no `onKeyDown`. There is no arrow-key (Up/Down/Left/Right) navigation and no Home/End for month edges. Keyboard-only users must Tab through all 35–42 day buttons to reach a target.
- **Reasoning (adversarial verdict — isReal):** Genuine WCAG 2.1.1 "Keyboard Accessible" gap. The codebase already implements the correct pattern elsewhere (`dropdownInternals.tsx`: `moveFocus()`, `focusEdge()`, `handleMenuKeyDown` with arrow support), so this is an implementation gap, not by-design. No parent mitigation (`FloatingPanel` only handles Escape + outside-click). Rated High (not Critical) because Tab + Enter/Space still works — it is UX degradation + a11y non-conformance, not total blockage.
- **Fix recommendation:** Add an `onKeyDown` handler on the Calendar wrapper that maintains a focused-day index in state, moves it on Arrow keys (wrapping across weeks), jumps to month edges on Home/End, and calls `onPick(day)` / `onHover(day)` accordingly. Mirror the focus-management approach in `dropdownInternals.tsx`. Apply the same change to the byte-identical template copy under `packages/create-luckystack-app/template/src/_components/inputs/`.

### Downgraded (no longer High/Critical)

- **DatePicker.updateTime ternary on line 268 — critical → MEDIUM.** `const startDay = which === "start" && selectedStart ? selectedStart : selectedStart;` returns `selectedStart` in both branches. Verdict `isReal=true` but downgraded: `startDay` is only consumed inside the `which === "start"` branch (where `selectedStart` is the correct value); the `which === "end"` path uses `startInstant` directly (line 276) and never reads `startDay`. The code is **behaviorally correct by accident** — this is a maintainability/clarity defect, not a runtime or security bug. Fix for clarity, not urgency.

### Refuted (false positives)

- **docs-ui `renderDocsScript` implicit-any embedded JS — HIGH → not-a-bug.** Adversarial verdict `isReal=false`. The finding conflates "JavaScript source embedded in a template string" with "TypeScript needing annotations." The embedded runtime script is intentionally self-contained (documented in CLAUDE.md JSX/self-contained conventions); the outer `renderDocsScript` is correctly typed (`string` params, `string` return). The cited `this.previousElementSibling` is deterministically safe given the adjacent template structure, and the finding references 3 non-existent functions (`renderApiGroup`, `renderAllApis`, `attachHandlers`), indicating a flawed audit. No security, null-safety, or runtime risk.
