Sources: reports/api.md + review/v0.2.0/* · Verified against current working tree (branch chore/package-split-prep, 2026-06-11).

## Verdict summary
13 distinct findings after de-duplication. Of these: 9 CONFIRMED, 0 already-fixed, 0 partially-fixed, 2 REFUTED-adjacent (none fully refuted — see notes), 2 UNCERTAIN-free. The two scans agree strongly and the `reports/` adversarial pass (CONFIRMED on H1 and H2) holds up against current code. The two biggest live issues are both still present and correctly rated: **runtime input validation is a no-op in production** (H1/SEC-02, high) and the **per-route rate limit is bypassable by rotating an unvalidated token** (H2, high), with two enforcement-correctness companions: a **raw session token leaked into the `rateLimitExceeded` hook + dev log** (SEC-01, high — the hook type literally promises "no tokens"), and an **unhandled-rejection socket hang on a null-user `additional` route** (M1, medium). The HTTP loopback rate-limit skip (M2) and the socket-only blanket eslint-disable (QUA-002) remain. Net: the package's two advertised enforcement guarantees (prod input validation, per-route rate limiting) are still not upheld.

## Findings

### F1 — Runtime input validation is a no-op in production · severity: high · status: CONFIRMED
- **Sources:** both — reports(H1) / review(SEC-02)
- **Current location:** `packages/core/src/runtimeTypeValidation.ts:305-307`; consumers `packages/api/src/_shared/socketValidationStage.ts:54-55`, `packages/api/src/_shared/httpValidationStage.ts:38`
- **Original claim:** `validateInputByType` short-circuits with `if (process.env.NODE_ENV === 'production') return { status: 'success' }` before any validation, so every prod endpoint accepts arbitrary shapes (type-confusion / Prisma operator injection). Docs advertise validation as a live boundary.
- **Verification (current code):** Confirmed verbatim. `runtimeTypeValidation.ts:305-307` returns `{ status: 'success' }` unconditionally when `NODE_ENV === 'production'`, before the devkit type resolver is reached. Both transports call into it: the socket stage runs `validateInputByType` only in `'strict'` mode (`socketValidationStage.ts:49-60`) and the HTTP stage always (`httpValidationStage.ts:38`), but both bottom out at the same prod no-op. So in production a handler typed `data: { userId: string }` is invoked with whatever JSON the caller sent. The "don't echo the raw validator message" security comments (`socketValidationStage.ts:71-75`) imply a live boundary that does not exist in prod.
- **Verdict & why:** CONFIRMED, high. reports/ rated High, review/ filed it under pkg-core but the consumer is the api pipeline; severity agrees. The dev-only intent is deliberate (avoid loading tsc in prod), but the security consequence — no runtime shape enforcement in the one environment that matters — is real and reachable on any public route. review/ additionally caught that `devkit/docs/runtime-type-resolver.md` *falsely claims* prod uses pre-generated Zod schemas while no request-time consumer of `apiInputSchemas` exists — a doc lie on top of the gap.
- **Recommendation:** Wire the already-generated Zod schemas (`apiInputSchemas.generated.ts`) into the prod path: on `NODE_ENV === 'production'`, look up the route schema and `safeParse`, succeeding only when no schema exists. Until then, correct the false doc claim and add a loud caveat to `ARCHITECTURE_API.md` + the package CLAUDE.md (which still advertises "input validation (Zod from generated types)" as a pipeline stage with no prod caveat).

### F2 — Per-route rate limit keyed on the unvalidated raw token (anonymous bypass) · severity: high · status: CONFIRMED
- **Sources:** reports(H2)
- **Current location:** `packages/api/src/handleApiRequest.ts:131-133`; `packages/api/src/handleHttpApiRequest.ts:302-304`
- **Original claim:** `requesterIdentity = token ?? resolvedIp` with `keyPrefix = token ? 'token' : 'ip'`. `token` is the raw cookie/Bearer value, never required to resolve to a real session, so an anonymous caller rotating a fresh random token per request lands each in a new per-route bucket, defeating `apiEntry.rateLimit` / `defaultApiLimit`.
- **Verification (current code):** Confirmed. Socket: `handleApiRequest.ts:131` `const requesterIdentity = token ?? resolvedIp;` → key `${keyPrefix}:${requesterIdentity}:api:${resolvedName}`. HTTP: `handleHttpApiRequest.ts:302` `const requesterIdentity = token ?? requesterIp ?? 'anonymous';` → identical key construction. `readSession(token)` (socket :372, http :146) may return null and the key still uses the raw token. `extractTokenFromRequest`/handshake token is fully attacker-controlled. The per-route bucket is the primary abuse control for expensive public routes.
- **Verdict & why:** CONFIRMED, high. reports/ adversarial pass already CONFIRMED this and the verdict holds. The global per-IP `:api:all` bucket still bounds total abuse in production (loopback skip is gated on non-prod — see F5), but the *per-route* guarantee is bypassable exactly as described.
- **Recommendation:** Key the per-route bucket on the validated `user.id` when a session resolved, and on the resolved IP otherwise — never on the raw unvalidated token. This also resolves F3 (the token would no longer be in the key). Add a `resolveRateLimitIdentity` hook/config callback (see F11) so consumers can key by API-key/tenant.

### F3 — Raw session token leaked into `rateLimitExceeded` hook payload + dev log (contract says "no tokens") · severity: high · status: CONFIRMED
- **Sources:** review(SEC-01)
- **Current location:** `packages/api/src/handleApiRequest.ts:144` (hook `key`), `:152` (dev log); mirror `packages/api/src/handleHttpApiRequest.ts:313-314`, `:322-323`; contract `packages/core/src/hooks/types.ts:205-206`; redaction set `packages/core/src/redactedLogKeys.ts:10-18`
- **Original claim:** The per-route key `token:<token>:api:<route>` is passed verbatim into `dispatchHook('rateLimitExceeded', { key: rateLimitKey, ... })` and logged via `getLogger().warn(..., { key: rateLimitKey })`, while the payload type documents `key` as "sanitized — no tokens" and `key` is not in `DEFAULT_REDACTED_LOG_KEYS`.
- **Verification (current code):** Confirmed. `handleApiRequest.ts:142-150` dispatches `rateLimitExceeded` with `key: rateLimitKey` where `rateLimitKey` is `${keyPrefix}:${requesterIdentity}:api:${resolvedName}` and `requesterIdentity = token` for authenticated callers (line 131-133). Line 152 logs `{ route: resolvedName, key: rateLimitKey }`. The HTTP mirror is identical (`:313` hook, `:323` log). `hooks/types.ts:205` still reads `/** The key that exceeded its limit (sanitized — no tokens). */`. `redactedLogKeys.ts:10-18` does not include `key` (and could not redact a token embedded *inside* a string value anyway).
- **Verdict & why:** CONFIRMED, high. review/ rated this High and it is correct: consumers are explicitly told the field is safe to persist to audit logs / external alerting sinks, turning a live session token into an account-takeover primitive in those sinks. Distinct root-cause surface from F2 but the same line-131 token-in-key is the source; fixing F2 (key on userId, not token) also fixes this.
- **Recommendation:** Pass a sanitized key to the hook + log — substitute the userId (`token:user:<userId>:api:<route>`) or hash/truncate the token. Keep the raw token only for the internal `checkRateLimit` call.

### F4 — `validateRequest` throws on null user for a public `additional` route → socket request hangs (unhandled rejection) · severity: medium · status: CONFIRMED
- **Sources:** reports(M1) — related code-quality note reports(§ validateRequest typing)
- **Current location:** `packages/core/src/validateRequest.ts:69`; reached from `packages/api/src/handleApiRequest.ts:87` `validateRequest({ auth: apiEntry.auth, user: user! })`; call site `packages/server/src/loadSocket.ts:148` `void handleApiRequest({ msg, socket, token })`
- **Original claim:** For `auth: { login: false, additional: [...] }`, an anonymous request has `user === null`. `checkApiAuth` skips the login gate and calls `validateRequest(user!)` = null; `condition.key in user` throws `TypeError: Cannot use 'in' operator ... in null`. The socket call site is fire-and-forget with no catch, so no response is ever emitted and the client hangs. HTTP survives (wrapped in `tryCatch`).
- **Verification (current code):** Confirmed on every link. `validateRequest.ts:69` `if (!(condition.key in user))` with `user` typed non-null `BaseSessionLayout` but receiving `null` via the `user!` assertion at `handleApiRequest.ts:87`. `checkApiAuth` is not wrapped in `tryCatch` and `applyApiRateLimits`/auth run synchronously inside the un-caught `handleApiRequest` promise. `loadSocket.ts:148` is `void handleApiRequest({ msg, socket, token })` — no `.catch`. The HTTP twin `apiRoute.ts:34` wraps the whole call in `tryCatch` and returns 500, so only the socket transport hangs. The HTTP handler's own `validateRequest({ auth, user: user! })` at `handleHttpApiRequest.ts:279` has the same null-pass but is caught.
- **Verdict & why:** CONFIRMED, medium. Triggered by a plausible misconfiguration (public route + `additional` predicate) and yields a transport-divergent availability bug (socket hang vs HTTP 500). Note `validateRequest` returns `auth.invalidCondition` (httpStatus 500) for a *missing* key but only after the `in` check — the throw happens first when `user` is null, because `'x' in null` throws before any branch.
- **Recommendation:** Have `validateRequest` treat a null/undefined user as `auth.forbidden` (or `auth.invalidCondition`) instead of throwing, and fix its `user: BaseSessionLayout` signature to `BaseSessionLayout | null` so the hole is visible to tsc. Optionally wrap `checkApiAuth` so any thrown auth check becomes a clean error envelope, and/or give the socket call site a `.catch`.

### F5 — Non-production loopback skip disables the global per-IP rate limit behind a localhost reverse proxy · severity: medium · status: CONFIRMED
- **Sources:** reports(M2)
- **Current location:** `packages/api/src/handleHttpApiRequest.ts:343-346`
- **Original claim:** `requesterIsLoopback = NODE_ENV !== 'production' && (requesterIp is 127.* / ::1 / ::ffff:127.*)`. With default `http.trustProxy: false`, `requesterIp` is `req.socket.remoteAddress` = `127.0.0.1` for every request when a reverse proxy terminates on the same host, so any non-prod deploy behind a local proxy skips the `:api:all` bucket for all clients. Combined with F2 this leaves a non-prod node with effectively no API rate limiting.
- **Verification (current code):** Confirmed verbatim at `handleHttpApiRequest.ts:343-346`; the guard wraps the entire global-IP bucket (`if (!requesterIsLoopback && defaultIpLimit ...)` at :346). `apiRoute.ts:67-71` resolves `requesterIp` from `req.socket.remoteAddress` under default `trustProxy: false`, confirming the collapse-to-127.0.0.1 premise behind a same-host proxy. Socket transport has NO equivalent skip (`applyApiRateLimits` always applies the global bucket), confirming the reports/ observation that the skip is HTTP-only and divergent.
- **Verdict & why:** CONFIRMED, medium. The skip is intentional (let the localhost test runner hammer endpoints) but infers "is loopback" from `NODE_ENV` + a proxy-collapsed/spoofable address. Staging / `NODE_ENV` unset / preview behind a local proxy silently lose the cross-route limit.
- **Recommendation:** Gate the skip on an explicit opt-in config flag (e.g. `rateLimiting.skipLoopbackInDev`, default false) rather than inferring from `NODE_ENV` + address. (See F11 — no config toggle exists today.)

### F7 — `as unknown as` double-cast in the API transport handlers · severity: medium · status: PARTIALLY-FIXED
- **Sources:** review(QUA-018)
- **Current location:** `packages/api/src/handleApiRequest.ts:396-397` (still bare, hidden by blanket disable); `packages/api/src/handleHttpApiRequest.ts:121` (now annotated + per-line disabled)
- **Original claim:** Both handlers do `normalized as unknown as Record<string, unknown> & { status?: string }` to feed `applyErrorFormatter`; root cause is `applyErrorFormatter` accepting a wider shape than the envelope union structurally satisfies. Zero-tolerance casting policy makes these reportable.
- **Verification (current code):** The cast still exists in BOTH files. `handleApiRequest.ts:397` `response: normalized as unknown as Record<string, unknown> & { status?: string }` — bare, masked by the file's blanket eslint-disable (F6). `handleHttpApiRequest.ts:121` `const formatterInput = transformPayload.response as unknown as Record<string, unknown> & { status?: string };` — now preceded by an explanatory comment (`:116-119`) and `// eslint-disable-next-line no-restricted-syntax -- formatter boundary cast`. So the *defect* (double-cast) persists in both; only the HTTP one is now documented/justified rather than hidden.
- **Verdict & why:** PARTIALLY-FIXED. The HTTP cast was converted from a hidden cast to a documented framework-boundary cast (acceptable under the "single typed adapter / annotated boundary" carve-out); the socket cast remains hidden and unannotated. Root cause (formatter input type too wide) is unaddressed in both.
- **Recommendation:** Widen `applyErrorFormatter`'s input in `@luckystack/core` to accept the envelope union directly (export a shared `ApiResponseEnvelope`), eliminating both casts. If a runtime boundary truly remains, keep a single annotated adapter — and at minimum annotate the socket cast like the HTTP one.

### F8 — `validation: 'relaxed'` / `{ input: 'skip' }` honored on socket but ignored on HTTP · severity: medium · status: CONFIRMED
- **Sources:** reports(Hard blocks)
- **Current location:** `packages/api/src/_shared/httpValidationStage.ts:17-19` + `:38` (always validates); `packages/api/src/_shared/socketValidationStage.ts:22-26`, `:47` (honors the flag)
- **Original claim:** The documented use case for relaxed validation is public webhooks (Stripe/Slack/GitHub) — which arrive over HTTP, the exact transport that ignores the flag. A webhook route whose third-party payload can't be modeled in TS is unreachable over HTTP in dev (strict validation 400s it), with no per-route escape hatch.
- **Verification (current code):** Confirmed. `socketValidationStage.ts` has `resolveValidationMode` (`:22-26`) and skips `validateInputByType` when mode is `'relaxed'` (`:49`, `:83-92`). `httpValidationStage.ts` has NO mode resolution and always calls `validateInputByType` (`:38`); its own header comment (`:17-19`) states "unlike the socket handler this stage always validates — the HTTP transport does not honor `validation: 'relaxed'`". Behaviour preserved verbatim during the god-function decomposition. Note F1 masks this in *production* only (validation globally off there), so the route works in prod and 400s in dev — the worse-than-symmetric failure reports/ flagged.
- **Verdict & why:** CONFIRMED, medium. Structural inconsistency between transports for a first-class documented feature, biting exactly the documented use case.
- **Recommendation:** Port `resolveValidationMode` into the HTTP stage and skip validation in relaxed mode (same as socket). Document the (now-symmetric) behavior in the package CLAUDE.md, which currently only notes relaxed is "skippable on socket transport."

### F9 — No `apiAuthRejected` hook on the API request lifecycle · severity: medium · status: CONFIRMED
- **Sources:** review(HOK-04)
- **Current location:** `packages/api/src/handleApiRequest.ts:70-105` (`checkApiAuth`), `packages/api/src/handleHttpApiRequest.ts:257-293`; hook map `packages/core/src/hooks/types.ts:300-324`
- **Original claim:** `checkApiAuth` and the HTTP equivalent emit `auth.required` (401) / `auth.forbidden` (403) but dispatch no hook on denial, while every other rejection class (`rateLimitExceeded`, `csrfMismatch`, `corsRejected`) has an observability hook. No `apiAuthRejected`/`authFailed` in `HookPayloads`.
- **Verification (current code):** Confirmed. `checkApiAuth` (socket :76-104) and the HTTP auth block (:258-293) only call `emitApiError`/`buildNetworkError` with no `dispatchHook` on the auth-fail path. `HookPayloads` (`hooks/types.ts:300-324`) lists `rateLimitExceeded`, `corsRejected`, `csrfMismatch` but no auth-rejection entry; `RateLimitExceededPayload.scope` even includes an `'auth'` literal (`:204`) that nothing emits. So a consumer cannot audit failed authorization / detect credential-stuffing on protected routes without forking.
- **Verdict & why:** CONFIRMED, medium. Genuine missing observability seam, consistent with the framework's own pattern for every other rejection class.
- **Recommendation:** Add an `apiAuthRejected` hook fired from both transports on the auth-fail path, shaped like `rateLimitExceeded` (`{ routeName, reason: 'login-required' | 'forbidden', errorCode, userId?, transport, ip? }`), dispatched via `void dispatchHook(...)` so it can't block the rejection.

### F10 — Socket/HTTP `data` shape guards accept an array · severity: low · status: CONFIRMED
- **Sources:** reports(L1)
- **Current location:** `packages/api/src/handleApiRequest.ts:59` (`typeof data != 'object'`); `packages/api/src/handleHttpApiRequest.ts:205` (`typeof data !== 'object' || data === null`)
- **Original claim:** `typeof [] === 'object'`, so `data: []` passes the guard and is cast to `Record<string, unknown>`. The HTTP guard has the same gap (only excludes null).
- **Verification (current code):** Confirmed. `handleApiRequest.ts:59` `if (!name || !data || typeof name != 'string' || typeof data != 'object')` — an array is truthy and `typeof === 'object'`, so it passes and is cast `data as Record<string, unknown>` at :67. `handleHttpApiRequest.ts:205` `if (typeof data !== 'object' || data === null)` — rejects null but not arrays. Low impact (handlers index by key) but the shape contract is looser than the type implies; combined with F1 (no prod validation) an array payload reaches handlers unchecked in prod.
- **Verdict & why:** CONFIRMED, low. Real but minor; matters more once F1 is fixed (validator would catch it) or stays as defense-in-depth.
- **Recommendation:** Reject arrays explicitly in both guards (`Array.isArray(data)` check).

### F11 — Backpressure poll interval + threshold hardcoded; no config for rate-limit identity or loopback skip · severity: low · status: CONFIRMED
- **Sources:** both — reports(Missing config / Hooks) / review(CFG-29)
- **Current location:** `packages/api/src/_shared/backpressure.ts:18` (1 MB default threshold), `:34` (`setTimeout(resolve, 10)` poll); identity/loopback have no knob anywhere in `handleApiRequest.ts` / `handleHttpApiRequest.ts`

- **Verification (current code):** Confirmed. `backpressure.ts:16-18` defaults `effectiveThresholdBytes` to `1_048_576` (overridable via `thresholdBytes`), and `:34` polls with a fixed `setTimeout(resolve, 10)` — no `pollIntervalMs` param or config read. - **Verdict & why:** CONFIRMED, low. review/ scoped this to just the backpressure poll (CFG-29); reports/ added the (more security-relevant) missing rate-limit-identity hook and loopback toggle.
