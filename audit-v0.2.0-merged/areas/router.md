# router — Verified & Merged Audit Findings
Sources: reports/router.md + review/v0.2.0/* · Verified against current working tree (branch chore/package-split-prep, 2026-06-11).

## Verdict summary
12 distinct findings merged across the two scans. None CONFIRMED at high/critical with a clean exploit: the single high-severity claim (SEC-08 / H1 — router forwards client-controlled `X-Forwarded-For`) is **REFUTED** as a router defect — `reports/` already did this adversarial pass and current code confirms it (backend `http.trustProxy` defaults to `false`, so XFF is ignored in the shipped config; the router is documented as an internal hop behind a TLS-terminating edge proxy that owns XFF sanitization). The review/ scan, which has no verification pass, over-rated it as High. Everything else is genuine but lower-severity hardening/extensibility/config gaps that are all still present in current code (CONFIRMED): no upstream proxy timeout (M2/SEC-30), unauthenticated+unnamespaced Redis health channel (M1), `redisHealthStore` ignores `REDIS_USER` (CFG-18), WS target service unconfigurable through the supported entry point (CFG-19), `console.error` instead of `getLogger()` (QUA-076), observe-only `preProxyRequest` hook (HOK-13), no hook on routing-rejection/WS paths (HOK-25/MIS-025), no router liveness/readiness endpoint (MIS-015), and the documented L1 absolute-URL defense-in-depth gap. The biggest live issue is **SEC-30/M2** (no upstream timeout — slow-loris/connection-exhaustion vector on an internet-facing router), followed by **M1** (shared-Redis health-channel poisoning). Nothing appears to have been fixed by commit 302cbf1 in this package — the router code is unchanged from what both scans describe; the scans simply differ on the severity of SEC-08.

## Findings

### SEC-08 / H1 — Router never sets/sanitizes `X-Forwarded-For` (client IP spoofing / rate-limit bypass) · severity: high (review) / refuted (reports) · status: REFUTED
- **Sources:** both — review(SEC-08, high) / reports(H1, refuted)
- **Current location:** `packages/router/src/httpProxy.ts:67-74`; mirror `packages/router/src/wsProxy.ts:43-52`; backend `packages/core/src/resolveClientIp.ts:78-97`, `packages/server/src/httpRoutes/apiRoute.ts:67-71`, default `packages/core/src/projectConfig.ts:464`
- **Original claim:** The proxy forwards client headers verbatim (`stripHopByHopHeaders` keeps XFF) and only adds host/proto/env headers, never overwriting `x-forwarded-for`. Backend trusts leftmost XFF when `trustProxy: true`, so a client can spoof its IP and defeat per-IP rate limiting + poison audit logs.
- **Verification (current code):** Code facts confirmed exactly: `httpProxy.ts:67-74` spreads `stripHopByHopHeaders(req.headers, ...)` (XFF is not hop-by-hop, passes through) and adds only `x-forwarded-host`/`x-forwarded-proto`/`x-luckystack-resolved-env`/`x-luckystack-via-fallback`; `wsProxy.ts:43-52` identical. `resolveClientIp.ts:83` returns the raw socket peer and ignores ALL headers when `trustProxy` is falsy; `:85-90` takes the leftmost XFF only when `trustProxy` is true. `apiRoute.ts:67` reads `getProjectConfig().http.trustProxy`, which defaults to `false` (`projectConfig.ts:464 trustProxy: false`). So in the shipped/default config the spoof is inert.
- **Verdict & why:** REFUTED as a router defect — reports/ is right, review/ over-rated severity. Three independent reasons: (1) `trustProxy` is OFF by default, so XFF is ignored. (2) The router's own `CLAUDE.md` ("does not terminate TLS… Front it with the platform's TLS-terminating proxy") positions it as an internal hop behind nginx/ALB/Caddy — that edge proxy is the component responsible for sanitizing/setting XFF, and the router correctly forwards the already-trusted value. (3) review/'s recommended fix (overwrite XFF with `req.socket.remoteAddress`) would actively break that documented topology by clobbering the real client IP the edge proxy populated. `resolveClientIp.ts:25-26` and `projectConfig.ts:185` already warn that `trustProxy: true` without a sanitizing proxy lets clients spoof — i.e. this is an operator-gated, documented configuration tradeoff, not a reachable router vulnerability.
- **Recommendation:** No code change. Optionally a one-line note in the router docs that, behind the router, the edge proxy (not the router) must set/sanitize XFF before `trustProxy: true` is enabled on the backend — but this is already covered by the core warnings.

### SEC-30 / M2 — No upstream/proxy timeout (slow-loris / connection-exhaustion) · severity: medium · status: CONFIRMED
- **Sources:** both — review(SEC-30, medium) / reports(M2, medium)
- **Current location:** `packages/router/src/httpProxy.ts:59-75` (HTTP), `packages/router/src/wsProxy.ts:35-53` (WS)
- **Original claim:** `transport.request({...})` sets no `timeout` and the code never calls `forwardRequest.setTimeout(...)`/`socket.setTimeout(...)`; a hung/slow upstream (or slow-read client) holds client↔router↔upstream sockets open indefinitely. No config knob to bound it. The only timeout (`healthPoller.ts:32`, 2s) is dev-gated.
- **Verification (current code):** Confirmed. `httpProxy.ts:59` `const forwardRequest = transport.request({ hostname, port, path, method, headers }, ...)` — no `timeout` option, no `setTimeout` call anywhere in the file; body is `req.pipe(forwardRequest)` (`:134`) with no body-size cap. `wsProxy.ts:35` `transport.request({...})` identical, no timeout. `healthPoller.ts:32-43` is the only timeout and `startRouter.ts:142` gates the poller to `isDevMode && enableFallbackRouting`, so it never runs in production.
- **Verdict & why:** CONFIRMED. Genuine hardening gap on an internet-facing edge process. Both scans agree on medium; that is the right call (no auth bypass, but a real resource-exhaustion vector with zero mitigation knob).
- **Recommendation:** Add a configurable `deploy.routing.upstreamTimeoutMs` (default ~30000) wired into `transport.request({ timeout })` + a `forwardRequest.on('timeout', ...)` that destroys the socket and emits the existing `routing.upstreamUnreachable` 502; consider `deploy.routing.maxRequestBodyBytes` for the unbounded `req.pipe`.

### M1 — Redis health channel/keys unauthenticated and unnamespaced (cross-tenant routing-health poisoning) · severity: medium · status: CONFIRMED
- **Sources:** reports(M1, medium) — (review/ has no direct equivalent)
- **Current location:** `packages/router/src/redisHealthStore.ts:31-35` (key/channel shape), `:54-65` (subscribe + apply)
- **Original claim:** Channel `router:health:events:<envKey>` / key `router:health:<envKey>:<service>` carry no project namespace (bypass core's `formatKey`/`registerRedisKeyFormatter`), no sender authentication, and `parsed.service` is not constrained to locally-owned services. Any Redis publisher (co-tenant on shared Redis) can flip a service healthy/unhealthy or cross-contaminate two deployments sharing one Redis + envKey.
- **Verification (current code):** Confirmed. `healthKey`/`healthChannel` (`:31-35`) build raw `router:health:...` strings with no `formatKey`/project prefix. `subscriber.on('message', ...)` (`:55-65`) validates only JSON shape (`typeof parsed.service === 'string'`, `typeof parsed.healthy === 'boolean'`) then unconditionally `cache.set(parsed.service, parsed.healthy)` + `input.onExternalChange(...)` — no check that `service` is in `getLocallyOwnedServices()`, no sender auth. Contrast `formatKey`/`registerRedisKeyFormatter` in core which every other framework key routes through.
- **Verdict & why:** CONFIRMED. Real on a shared/multi-tenant Redis (a topology the framework explicitly supports). Medium is right — exploit requires Redis `PUBLISH` access, not raw network reach.
- **Recommendation:** Route these keys/channels through the core key formatter (project prefix) so two deployments can't collide, and validate `parsed.service` against `resolver.getLocallyOwnedServices()` before applying (treat external health as advisory).

### CFG-18 — `redisHealthStore` ignores `REDIS_USER`, cannot auth to ACL Redis · severity: medium · status: CONFIRMED
- **Sources:** review(CFG-18, medium) — overlaps reports/ "missing config" + "code quality" notes
- **Current location:** `packages/router/src/redisHealthStore.ts:40-45`; correct pattern at `packages/router/src/bootHandshake.ts:98`; core source at `packages/core/src/redis.ts:68-73`
- **Original claim:** `createRedisHealthStore` builds ioredis clients from raw env (`REDIS_HOST`/`REDIS_PORT`/`REDIS_PASSWORD`) and never reads `REDIS_USER`, diverging from `bootHandshake.ts:98` which uses `getRedisConnectionOptions()`. On Redis 6+ ACL the boot handshake authenticates but the health store fails (NOAUTH/WRONGPASS), which in split/fallback mode hard-fails startup.
- **Verification (current code):** Confirmed. `redisHealthStore.ts:40-42` reads `process.env.REDIS_HOST`/`REDIS_PORT`/`REDIS_PASSWORD` directly; `:44-45` `new Redis({ host, port, password, lazyConnect: true })` — no `username`. `bootHandshake.ts:98` does `getRedisConnectionOptions()`, and `core/src/redis.ts:68-73` shows that helper DOES include `username` from `REDIS_USER` (`...(process.env.REDIS_USER ? { username: process.env.REDIS_USER } : {})`). The two router Redis paths have drifted. `startRouter.ts:92-100` calls `createRedisHealthStore` with only `{ envKey, onExternalChange }`, so the optional `redisHost`/`redisPort`/`redisPassword` overrides can't compensate, and there is still no `username` override field at all.
- **Verdict & why:** CONFIRMED. Also a DRY/SOLID violation (connection-option logic duplicated instead of sourced from core). Medium is right — only bites ACL-username Redis deploys, but there it hard-fails startup in split/fallback mode.
- **Recommendation:** Replace the raw-env client construction with `getRedisConnectionOptions()` (spread `{ host, port, username?, password? }`), matching `bootHandshake`. Note the package `CLAUDE.md` "Config keys consumed" already (incorrectly) claims Redis connection is "read centrally via getRedisConnectionOptions()" — the doc is currently wrong for the health store; this fix makes it true.

### CFG-19 — WS target service pinned to `system`, no config/CLI override through the supported entry point · severity: medium · status: CONFIRMED
- **Sources:** both — review(CFG-19, medium) / reports("hard blocks" + "missing config" + "hooks")
- **Current location:** `packages/router/src/wsProxy.ts:14,18-22`; call site `packages/router/src/startRouter.ts:138`; no field in `StartRouterInput` (`startRouter.ts:27-54`) or CLI (`cli.ts:28-99`)
- **Original claim:** `createWsProxy` accepts a `wsTargetService` param but `startRouter` calls `createWsProxy({ resolver })` without threading it; `StartRouterInput` exposes no field, there is no `deploy.routing` key, and the CLI has no flag. A consumer whose socket service isn't literally named `system` gets a 502 on every WS upgrade with no fix short of forking.
- **Verification (current code):** Confirmed. `wsProxy.ts:14` `const DEFAULT_WS_SERVICE = 'system'`; `:21-22` `createWsProxy = ({ resolver, wsTargetService }) => { const service = wsTargetService ?? DEFAULT_WS_SERVICE; ...}` — the override exists but `startRouter.ts:138` `const wsProxy = createWsProxy({ resolver });` passes only `resolver`. `StartRouterInput` (`startRouter.ts:27-54`) has no `wsTargetService`/`websocketService` field; `cli.ts` `CliArgs` (`:28-35`) has no flag. The single-binding-per-service WS model is documented in `ARCHITECTURE_MULTI_INSTANCE.md`, but the service NAME being fixed is not surfaced as a tunable.
- **Verdict & why:** CONFIRMED. Both scans agree (reports/ frames it as a "hard block" + extensibility gap; review/ as a config gap). Medium is right.
- **Recommendation:** Thread `wsTargetService` through `StartRouterInput` and add `deploy.routing.websocketService?: string` (default `'system'`) consumed in `startRouter` when constructing the WS proxy + a `--ws-service` CLI flag; document alongside the other `routing.*` knobs.

### MIS-015 — Router process exposes no liveness/readiness endpoint · severity: medium · status: CONFIRMED
- **Sources:** review(MIS-015, medium) — (reports/ did not raise it)
- **Current location:** `packages/router/src/startRouter.ts:137-140`; rejection paths `httpProxy.ts:26-43`
- **Original claim:** `startRouter` does `http.createServer(proxy)` where every request goes straight into the proxy (`/` → 400 invalidRequestPath; `/_health` → parses `_health` as a service → 502). `@luckystack/server` ships `/livez`/`/readyz`/`/_health` but the router is a separate process with none, so k8s/Cloud Run/ECS can't probe the router itself without proxying through to a backend.
- **Verification (current code):** Confirmed. `startRouter.ts:139` `const server = http.createServer(proxy);` with `proxy = createHttpProxy(...)` (`:137`) — there is no pre-resolution router-local path. In `httpProxy.ts:18-43` every request runs `resolveServiceKey` then `resolver.resolve`; `/` yields `routing.invalidRequestPath` 400, and any non-routable path yields the 502 `missingServiceErrorCode`. No `/livez`/`/readyz`/router-health short-circuit exists.
- **Verdict & why:** CONFIRMED. Legitimate operability gap for a process meant to run as its own container. Medium is right (no security impact; affects deployability/observability).
- **Recommendation:** Reserve a router-local path (e.g. `deploy.routing.routerHealthPath`, default `/_router/health`) handled in `createHttpProxy`/`startRouter` before service resolution, returning 200 `{ status, env, bootUuid, healthStore: 'redis'|'in-memory' }`.

### HOK-13 / hooks — `preProxyRequest` is observe-only; cannot block or mutate the request · severity: medium · status: CONFIRMED
- **Sources:** both — review(HOK-13, medium) / reports("hooks & extensibility" + "docs gaps")
- **Current location:** `packages/router/src/httpProxy.ts:49-57` (dispatch), `:59` (upstream proceeds regardless)
- **Original claim:** `preProxyRequest` is `void dispatchHook(...)` fire-and-forget and the upstream request proceeds immediately; the payload carries no mutable handle to headers/target and there is no await/short-circuit. A consumer running the router as an edge LB cannot do IP/geo banning, maintenance-mode short-circuit, a WAF rule, or edge auth — despite the `CLAUDE.md` advertising "intercept proxy traffic … redaction" and "redact path segments."
- **Verification (current code):** Confirmed. `httpProxy.ts:51` `void dispatchHook('preProxyRequest', { service, pathname, method, target, viaFallback })` — not awaited, return value discarded; `forwardRequest = transport.request(...)` runs unconditionally at `:59`. Payload (per `hookPayloads.ts` + the inline shape) is plain serializable data with no mutation handle. The package `CLAUDE.md` "When to USE" does say consumers can "intercept proxy traffic (tracing IDs, audit logging, redaction)" and the inline comment at `httpProxy.ts:49-50` says consumers "redact path segments" — which the observe-only hook cannot actually do.
- **Verdict & why:** CONFIRMED as an extensibility gap + a docs-oversell. Medium for the missing deny/mutate hook; the docs claim ("redaction") is misleading and should be corrected to "observability-only."
- **Recommendation:** Add an awaited, decision-returning gate hook (e.g. `proxyRequestGate` returning `{ action: 'allow' } | { action: 'deny', statusCode, body }`) dispatched before `transport.request`, reusing core's stop-signal hook pattern; keep `preProxyRequest` as the fire-and-forget observer. Separately, fix `CLAUDE.md`/the inline comment to drop the "redact path segments"/"redaction" claim.

### HOK-25 — Routing-rejection responses (400/502) fire no hook · severity: low · status: CONFIRMED
- **Sources:** review(HOK-25, low) — (reports/ did not separately raise it)
- **Current location:** `packages/router/src/httpProxy.ts:26-31` (400 invalidRequestPath), `:34-43` (502 missingServiceErrorCode)
- **Original claim:** When the path has no parseable service (400) or no binding resolves (502), neither `preProxyRequest` nor `postProxyResponse` fires; only requests that reach a resolved upstream are observable, so metrics/audit wired via the hook bus is blind to rejected/misrouted traffic.
- **Verification (current code):** Confirmed. `httpProxy.ts:26-31` and `:34-43` both `res.end(...)` and `return` before any `dispatchHook` call; `preProxyRequest` is first dispatched at `:51`, after both early returns. So 400/502 rejections emit no hook.
- **Verdict & why:** CONFIRMED. Low severity — observability completeness, not a security/correctness defect.
- **Recommendation:** Emit `postProxyResponse` (or a dedicated `proxyRequestRejected` hook) with statusCode 400/502 + a rejection-reason field on the two early-return paths.

### MIS-025 — WebSocket upgrades emit no proxy hooks · severity: low · status: CONFIRMED
- **Sources:** both — review(MIS-025, low) / reports("hooks & extensibility")
- **Current location:** `packages/router/src/wsProxy.ts:24-90`
- **Original claim:** `createWsProxy` dispatches no hooks at all (no pre/post, no connect/disconnect) — acknowledged in `packages/router/docs/post-proxy-response-hook.md:8` as a known limitation. A socket-first framework's edge gets zero router-level WS observability.
- **Verification (current code):** Confirmed. `wsProxy.ts:24-90` contains no `dispatchHook` call anywhere — no import of `dispatchHook` (unlike `httpProxy.ts:5`). The whole upgrade path (resolve → request → upgrade pipe → teardown) is hook-free.
- **Verdict & why:** CONFIRMED. Low — documented known gap, observability only.
- **Recommendation:** Add lightweight WS lifecycle hooks (`preWsUpgrade`/`postWsUpgrade` with `{ service, target, viaFallback, statusCode }`) mirroring the HTTP hook shape.

### QUA-076 — `console.error` instead of the registered logger · severity: low · status: CONFIRMED
- **Sources:** both — review(QUA-076, low) / reports("code quality")
- **Current location:** `packages/router/src/resolveTarget.ts:255-257`
- **Original claim:** The fire-and-forget health-publish failure handler logs via `console.error('[router] failed to publish health change:', error)` while the rest of the package routes through `getLogger()`. This one error class bypasses the consumer's structured/JSON/Datadog log pipeline. Note: `resolveTarget.ts` currently imports only types from core, so adding `getLogger` introduces a runtime import.
- **Verification (current code):** Confirmed. `resolveTarget.ts:256` `console.error('[router] failed to publish health change:', error);` inside the `void healthStore.set(...).catch(...)` at `:255-257`. The file's imports (`:1-7`) are `tryCatchSync` + types only — confirming the noted runtime-import cost. By contrast `startRouter.ts`/`bootHandshake.ts`/`healthPoller.ts` use `getLogger()`.
- **Verdict & why:** CONFIRMED. Low — consistency/log-pipeline issue, not a defect.
- **Recommendation:** Import `getLogger` and use `getLogger().error('[router] failed to publish health change', { message })`.

### L1 — Absolute-form request target not explicitly rejected (defense-in-depth) · severity: low · status: CONFIRMED
- **Sources:** reports(L1, low) — (review/ did not raise it)
- **Current location:** `packages/router/src/httpProxy.ts:46` (`new URL(pathname, resolved.target)`); guard side-effect `resolveTarget.ts:85-96`
- **Original claim:** `pathname` is the raw `req.url`; `new URL(absoluteUrl, base)` ignores the base when the first arg is absolute, so an attacker-controlled host could in principle become `targetUrl.hostname`/`port`. Currently mitigated only incidentally because `parseServiceFromPath` extracts `http:`/empty as the "service" for absolute/`//` targets → `resolve(...)` returns null → 400/502 before the `new URL`. The safety rests on a parsing side effect, not an explicit guard.
- **Verification (current code):** Confirmed. `httpProxy.ts:46` `const targetUrl = new URL(pathname, resolved.target);` with `pathname = req.url ?? '/'` (`:19`). `resolveTarget.ts:85-96` `parseServiceFromPath` does `parseFirstSegment`, which for an absolute URL like `http://evil/...` returns the first segment `http:` (or empty for `//`), so `resolver.resolve('http:')` returns null → the 400/502 fires at `httpProxy.ts:26-43` before line 46. So today it is not exploitable — but the URL construction itself still does not pin the host, exactly as claimed.
- **Verdict & why:** CONFIRMED as a latent defense-in-depth gap (reports/ rated it low and that is right). A future custom `registerServiceResolver` or host-based routing could re-open an SSRF path because the host pinning is incidental, not explicit.
- **Recommendation:** Reject non-origin-form targets (must start with `/`) before resolution, and construct the upstream URL from `resolved.target` + a validated path rather than letting `new URL` re-host.

### L2 — Health-store Redis clients have no `error` listener / reconnect policy · severity: low · status: CONFIRMED
- **Sources:** reports(L2, low) — (review/ did not raise it)
- **Current location:** `packages/router/src/redisHealthStore.ts:44-50`
- **Original claim:** `new Redis({ host, port, password, lazyConnect: true })` for both `client` and `subscriber` registers neither `.on('error', ...)` nor a `retryStrategy`. A post-boot Redis blip surfaces as unhandled ioredis `error` events (noisy / process-destabilizing) and silent loss of cross-router health propagation with no surfaced signal.
- **Verification (current code):** Confirmed. `redisHealthStore.ts:44-45` constructs both clients with `{ host, port, password, lazyConnect: true }` and no `.on('error')`/`retryStrategy`. After the initial `await client.connect()`/`await subscriber.connect()` (`:49-50`), a later connection error has no handler. (The `close` path uses `tryCatch` but that only covers teardown.)
- **Verdict & why:** CONFIRMED. Low — runtime robustness; relevant on a long-running router whose Redis blips post-boot.
- **Recommendation:** Attach an `error` handler that logs via `getLogger()` and surfaces a degraded-state signal; consider an explicit `retryStrategy`.

### CC — Duplicate Redis client for the boot compare · severity: low · status: CONFIRMED
- **Sources:** reports("code quality") — (review/ did not raise it)
- **Current location:** `packages/router/src/bootHandshake.ts:144-149`
- **Original claim:** `bootHandshake` opens a second `Redis` connection (`fallbackClient`) with the same `redisOptions` purely to `GET` one key it could have read on the already-open `client`. Minor waste; `localReadOfFallbackKey` name slightly obscures that it reads from the router's own Redis.
- **Verification (current code):** Confirmed. `bootHandshake.ts:100` opens `client`, uses it for the boot `set`, then `client.disconnect()` at `:107` — so by the time the compare runs (`:144-149`) the first client is already disconnected, which is why a second `fallbackClient` is opened. So the duplicate is partly explained by the early disconnect, not pure waste; the connection could instead be kept open until after the compare. Minor.
- **Verdict & why:** CONFIRMED but trivial. Low/cosmetic — correctness is fine.
- **Recommendation:** Keep the single `client` open until after the fallback-key `GET`, or leave as-is (the cost is one short-lived connection at boot). Not worth a change unless touching the file.

### CLI — Unknown CLI flags silently ignored · severity: low · status: CONFIRMED
- **Sources:** reports("code quality") — (review/ did not raise it)
- **Current location:** `packages/router/src/cli.ts:90-94`
- **Original claim:** The arg parser swallows unrecognized flags by design; a typo like `--no-shared-helth` silently leaves shared health on with no warning — a foot-gun for an ops-facing CLI.
- **Verification (current code):** Confirmed. `cli.ts:90-94` `default: { /* Ignore unknown flags ... */ break; }` — no warning emitted. A misspelled `--no-shared-health` would be ignored and shared health stays enabled.
- **Verdict & why:** CONFIRMED. Low — ergonomics/ops foot-gun, intentional but worth a stderr warning.
- **Recommendation:** Emit a `process.stderr.write('[luckystack-router] ignoring unknown flag: <flag>')` in the default branch (still non-fatal) so typos surface.
