Area: `packages/router/src` — multi-instance HTTP + WebSocket reverse proxy / load balancer.
Working tree: branch `chore/package-split-prep`, HEAD `302cbf1` + ~248 uncommitted fixes (this week's remediation pass).

## Verdict

Every wave-2 fixed-claim was re-verified against the CURRENT source (files opened, file:line cited). The verification HOLDS: the two router CRITICALs (WS listener-less crash #1, WS SSRF #2) and the dominant HIGHs (XFF/XFP spoof #8/#9/#23, HTTP req/res abort leak #6, non-101 leak #5, subscriber-connect FD leak #14, boot-handshake error-listener gap #13) are genuinely fixed in code with `proxyUtils.ts` centralizing the security invariants and `wsProxy.test.ts` + `redisHealthStore.test.ts` regressions present. CLI NaN (#15) and SIGINT double-shutdown (#20) fixed. Body-cap race (#7) gone because the whole body-cap capability was removed.

What remains OPEN is the long tail (mediums/lows that were always low-blast-radius) plus a small set of NEW wave-2 items — most important the **WS+HTTP upstream leg has no timeout** (HIGH, real, present on both proxies) and the **doc drift in `http-proxy.md`** that now actively misdescribes the proto handling.

The wave-1 "deny-gate" findings (#3 WS gate gap, #11 gate fail-open) are FALSE-POSITIVE against current code: no `proxyRequestGate`/`preWsUpgrade`/`postWsUpgrade` mechanism exists anywhere in the package (grep: 0 hits) — the baseline assumed a gate that never shipped. The underlying capability gap (no edge deny mechanism) is real but is a FEATURE/DEFERRED-DECISION, not a bug.
